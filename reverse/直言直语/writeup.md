## 3. 真正关键函数：`sub_402600`

结合辅助 WP 与本地样本复核，确认真正决定成功/失败的是：

```text
sub_402600
```

进入该函数后，可以看到非常明显的一段初始化常量：

```text
5b d6 d0 26 c8 dd 19 7e 6e 3e cb 16 91 7d ff af dd 76 64 b0 f7 e5 89 57 82 9f 0c 00 9e d0 45 fa
```

也就是 32 字节比较数组（按函数内顺序存放）。

但该函数一开始的伪代码看起来会很乱，因为存在多处相同模板的**花指令**。如果不处理，会严重干扰对逻辑的理解。

---

## 4. 花指令与关键子函数

脱花并整理后，核心子函数可还原为：

- `sub_402CA0`：RC4 KSA（初始化 S 盒）
- `sub_402E80`：RC4 PRGA / XOR 加解密

其逻辑本质就是标准 RC4：

### `sub_402CA0`
```c
for (i = 0; i < 256; ++i) {
    S[i] = i;
    T[i] = key[i % key_len];
}
for (j = 0; j < 256; ++j) {
    v = (T[j] + v + S[j]) % 256;
    swap(S[j], S[v]);
}
```

### `sub_402E80`
```c
for (i = 0; i < len; ++i) {
    x = (x + 1) % 256;
    y = (y + S[x]) % 256;
    swap(S[x], S[y]);
    data[i] ^= S[(S[x] + S[y]) % 256];
}
```

RC4 key 在函数里可以直接恢复出来：

```text
qwertyuiop
```

## 5. 关键误判点：不是直接把 32 字节明文丢进 RC4

这一题最容易犯的错，是把 RC4 解出的 32 字节直接当作最终 flag 内部内容。

### 错误方向
直接把 RC4 可逆得到的：

```text
f250e3d75820847d427f3af11a783379
```

当成最终输入，或者简单再外包 `flag{}`。

### 为什么错
因为在进入 RC4 之前，程序先调用了：

```text
sub_402AF0
```

这个函数**不只是长度检查**，它还会对输入做预处理。

---

## 6. `sub_402AF0` 的真实作用

本地汇编复核显示，`sub_402AF0` 会做如下几步：

### 6.1 长度校验
先计算：

```text
strlen(input) - 6 == 0x20
```

也就是必须满足：

```text
flag{ + 32字节 + }
```

因此：

- 不带 `flag{}` 会失败
- 内部不是 32 字节也会失败

### 6.2 跳过 `flag{`
它从：

```text
input + 5
```

开始拷贝 32 字节到临时缓冲区，也就是取出花括号内部内容。

### 6.3 原地倒序
随后它对这 32 字节执行首尾交换，循环 16 次，相当于：

```text
reverse(inner_32_bytes)
```

### 6.4 清空原缓冲区并写回
最后清空输入缓冲区，再把倒序后的 32 字节写回输入开头。

也就是说，真正送入 `sub_402E80` 做 RC4 的内容是：

```text
reverse(flag_inside)
```

这一步就是前面多次手输失败的根源。

## 7. 比较逻辑
`sub_402600` 在 RC4 后的比较循环是：

```c
for (i = 31; i >= 0; --i) {
    if (buf[i] != const_array[i])
        fail;
}
```

注意这里对比的是**预处理 + RC4 后的缓冲区**，而不是用户输入的原始花括号内容。

程序中存放的 32 字节数组（按比较顺序）为：

```text
5b d6 d0 26 c8 dd 19 7e 6e 3e cb 16 91 7d ff af dd 76 64 b0 f7 e5 89 57 82 9f 0c 00 9e d0 45 fa
```

十六进制展开即：

```c
unsigned char arr[32] = {
  0x5b,0xd6,0xd0,0x26,0xc8,0xdd,0x19,0x7e,
  0x6e,0x3e,0xcb,0x16,0x91,0x7d,0xff,0xaf,
  0xdd,0x76,0x64,0xb0,0xf7,0xe5,0x89,0x57,
  0x82,0x9f,0x0c,0x00,0x9e,0xd0,0x45,0xfa
};
```

## 8. 正确恢复最终输入

### 8.1 先求出送入 RC4 前应有的 32 字节
对上述数组用 key `qwertyuiop` 做 RC4 解密，可得：

```text
f250e3d75820847d427f3af11a783379
```

这是**经过 `sub_402AF0` 处理后**真正参与 RC4 的 32 字节。

### 8.2 反推出用户输入的花括号内容
由于 `sub_402AF0` 会先倒序，所以用户真正应该输入的 32 字节应为其反转：

```text
973387a11fa3f724d74802857d3e052f
```

### 8.3 加上外壳
最终正确输入：

```text
flag{973387a11fa3f724d74802857d3e052f}
```
